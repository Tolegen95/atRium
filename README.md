# Умный мониторинг комфорта

Full-stack сервис, который превращает поток технических сообщений от датчиков атриума NU (температура, освещение, шум) в понятный сервис: показывает текущее состояние, историю измерений с фильтрами, аналитику по дню и позволяет пользователям оставлять отчёты о комфорте.

Данные о температуре в атриуме/на улице, уровне освещения и шума публикуются в Telegram-канале [@test_temp_nu](https://t.me/test_temp_nu).

## Данные

`backend/result.json` — полная выгрузка истории канала [@test_temp_nu](https://t.me/test_temp_nu) через Telegram API (Telethon), сделанная скриптами из `backend/scripts/`. Канал ведёт историю с сентября 2023 года, и формат сообщений менялся несколько раз (числовая влажность → числовой люкс → текущий формат с категориями освещения/шума), плюс в канале встречаются посты с прогнозом погоды и служебные объявления.

Парсер (`app/seed.py`) сознательно отбирает только надёжно размеченные показания:
- сообщение должно содержать температуру с явным маркером 🌡 и явную метку «Atrium»/🏫 или «Outside»/🌆 (иначе это не показание датчика — пропускается, это отсекает прогнозы погоды и чаты);
- освещение и шум сохраняются только если совпадают с одной из известных категорий (`Dark/Dim/Normal brightness/Bright/Very bright`, `Very quiet/Quiet/Mild noise/Noisy/Very noisy`) — старые сообщения с числовым люксом/влажностью импортируются как показания температуры с `brightness/noise = null`, а не мусорной строкой;
- NaN-температуры отбрасываются;
- полные дубликаты (тот же timestamp, то же место, то же значение — канал изредка постил одно сообщение дважды) схлопываются в одну запись.

Итог: 23 067 показаний за период с 2023-09-20 по сегодняшний день (10 217 атриум + 12 850 улица). Температура в атриуме держится в диапазоне 10.6–33.7°C, на улице — от -34.1°C зимой до 38.4°C летом, что соответствует резко континентальному климату Астаны.

**Часовой пояс**: атриум физически находится в Астане, поэтому все `measured_at` хранятся как naive wall-clock время `Asia/Almaty` (а не UTC) — это учитывает и переход Казахстана на UTC+5 в марте 2024 года (до этого было UTC+6). Telegram отдаёт время сообщений в UTC; конвертация в местное время происходит один раз при импорте (`app/seed.py`, `scripts/tg_3_export.py`), поэтому фронтенду не нужно ничего пересчитывать — время в интерфейсе всегда совпадает с тем, что видно в самом Telegram-канале.

Чтобы не отдавать все показания одним запросом, `GET /api/readings` без фильтра даты по умолчанию возвращает последние 500 записей (параметр `limit`, максимум 5000) — при выборе конкретного дня фильтр отсекает нужный срез и лимит не влияет на результат.

## Как запустить локально

### Backend

```bash
cd backend
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

При первом запуске backend автоматически загрузит `backend/result.json` в SQLite (`backend/atrium.db`), если база пустая. Это же можно сделать вручную:

```bash
python -m app.seed
```

Backend поднимется на `http://localhost:8000` (Swagger-документация — `http://localhost:8000/docs`).

### Frontend

```bash
cd frontend
npm install
cp .env.example .env.local       # VITE_API_URL=http://localhost:8000
npm run dev
```

Frontend поднимется на `http://localhost:5173` (или другой порт, если занят).

### Обновление данных из Telegram-канала (опционально)

Подключение к Telegram в реальном времени не требуется по условиям кейса. `backend/result.json` — это статичный экспорт истории канала, уже готовый к использованию. Чтобы обновить его свежей историей через Telegram API (Telethon), нужны свои `api_id`/`api_hash` с [my.telegram.org](https://my.telegram.org):

```bash
cd backend
source venv/bin/activate
python scripts/tg_1_send_code.py <api_id> <api_hash> <phone>
python scripts/tg_2_complete_login.py <code_from_telegram> [2fa_password]
python scripts/tg_3_export.py test_temp_nu
```

Скрипт перезапишет `backend/result.json`. Файлы сессии (`tg_export.session`, `tg_login_state.json`) в git не попадают.

## Правила статуса и comfort score

Пороговые значения — наш собственный выбор, отражающий здравый смысл для учебного/рабочего пространства (не медицинская или официальная рекомендация):

**Температура** — та же 5-уровневая шкала, что и у освещения/шума (`Reading.temperature` → `temperature_label`, [`backend/app/comfort.py`](backend/app/comfort.py)), применяется и к атриуму, и к улице:

| Диапазон | Значение |
|---|---|
| < 16°C | Очень холодно |
| 16–20°C | Холодно |
| 20–25°C | Комфортно |
| 25–30°C | Жарко |
| ≥ 30°C | Очень жарко |

**Итоговая текстовая метка** (главная страница, объединяет температуру/шум/освещение) выбирается по приоритету:
1. температура вне диапазона «Комфортно» (см. таблицу выше) → соответствующая метка («Очень холодно»/«Холодно»/«Жарко»/«Очень жарко»)
2. noise ∈ {Noisy, Very noisy} → «Шумно»
3. brightness == Dark → «Темно»
4. иначе — по comfort score (см. ниже): ≥75 и тихо/умеренно шумно → «Подходит для учёбы»; ≥75 иначе → «Комфортно»; ≥55 → «Комфортно»; <55 → «Лучше выбрать другое место»

**Comfort score (0–100)** — собственная формула:

```
score = 100
      - temp_penalty       # 5 очков за каждый °C ниже 20°C, 6 очков за каждый °C выше 25°C
      - noise_penalty      # Very quiet/Quiet: 0, Mild noise: 8, Noisy: 22, Very noisy: 30
      - brightness_penalty # Dark: 15, Dim: 5, Normal/Bright: 0/5, Very bright: 15
score = clamp(score, 0, 100)
```

Измерение считается «некомфортным», если `comfort_score < 55`.

Логика реализована в [`backend/app/comfort.py`](backend/app/comfort.py).

## Реализованные обязательные функции

- Импорт `result.json` в SQLite через SQLAlchemy ORM (`app/seed.py`), автоматически при первом запуске или вручную
- Модель `Reading` (id, measured_at, location, temperature, brightness, noise)
- Главная страница: текущие температура/освещение/шум в атриуме и на улице, время последнего измерения, текстовая оценка состояния
- История измерений: таблица, фильтр по дате, по месту (атриум/улица), по уровню шума, сортировка по времени и по температуре
- Аналитика: мин/макс/средняя температура за день, лучший период для учёбы, самое прохладное и самое тихое время, самый жаркий период, количество некомфортных измерений, средняя разница температур атриум/улица, comfort score дня
- Отчёты пользователей: модель `Report` (id, created_at, category, comment, status), полный CRUD (создание, просмотр, редактирование, удаление)
- REST API с корректными HTTP-кодами (404 при отсутствии записи, 422 при невалидных данных), валидацией через Pydantic, JSON-ответами
- Frontend-состояния: загрузка, ошибка, пустой результат, успешное создание/изменение отчёта (тосты)
- Адаптивная вёрстка (ноутбук и телефон)

## Реализованные бонусные функции

- **Визуализация**: график температуры атриум/улица за выбранный день (recharts)
- **Comfort score**: собственная формула и её объяснение в этом README
- **UX**: светлая/тёмная тема с переключателем, цветовое кодирование статуса (зелёный/жёлтый/красный), dashboard для отчётов (фильтр по статусу, отметка «решено»/«открыть снова»), подтверждение перед удалением, уведомления об успешных действиях
- **REST API**: дополнительные фильтры (`noise`, `brightness`, `min_temperature`, `max_temperature`), сортировка по времени/температуре, эндпоинт `/api/readings/current` для текущего состояния
- **Обновление данных по кнопке**: кнопка «🔄 Обновить данные» на главной странице подтягивает последние ~20 сообщений с публичной веб-версии канала (`https://t.me/s/test_temp_nu`) без каких-либо учётных данных Telegram, парсит их той же логикой, что и `app/seed.py`, и добавляет только новые (дедупликация по времени+месту). Так интерфейс может показывать почти актуальные данные без полноценного live-подключения к Telegram API, которое кейсом не требуется.

## API

| Метод | Маршрут | Описание |
|---|---|---|
| GET | `/api/readings` | список измерений с фильтрами и сортировкой |
| GET | `/api/readings/{id}` | одно измерение |
| GET | `/api/readings/current` | последнее состояние атриума + текстовый статус |
| GET | `/api/summary?date=YYYY-MM-DD` | аналитика за день (по умолчанию — последний день с данными) |
| POST | `/api/refresh` | подтянуть новые показания с публичной веб-версии канала |
| GET | `/api/reports` | список отчётов (фильтр по статусу) |
| POST | `/api/reports` | создать отчёт |
| PATCH/PUT | `/api/reports/{id}` | изменить отчёт |
| DELETE | `/api/reports/{id}` | удалить отчёт |
